# 月度 Git 工作汇总规则

## 适用范围

本规则用于生成指定人员在 `D:\Projects` 下所有工作项目的月度 Git 提交统计文档。

## 目录规则

- 默认扫描根目录：`D:\Projects`。
- 默认输出目录：skill 根目录。
- 默认排除目录：`auto-git`。
- 只统计包含 `.git` 的 Git 仓库目录。
- 不主动执行 `git fetch --all`、`git pull`、`git checkout` 等会改变本地仓库状态的命令。

匹配范围：

## 时间范围规则

- 统计粒度为自然月。
- 起始时间：目标月份第一天 `00:00:00`。
- 结束时间：下个月第一天 `00:00:00`，左闭右开。
- 示例：`2026-04` 的统计范围为 `2026-04-01 00:00:00` 至 `2026-05-01 00:00:00`。

## 分支范围规则

- 遍历本地分支和远程分支。
- 等价范围：`--branches --remotes`。
- 不统计 stash 引用，除非后续明确需要。
- 同名本地分支和远程分支合并展示。

## 分支归类规则

同一 commit 可能同时存在于多个分支中，最终只归入一个主分支，避免重复统计。

优先级如下：

1. 远程分支优先。
2. 本地分支次之。
3. 默认分支兜底。

远程分支选择规则：

- 优先选择非 `HEAD` 的远程分支。
- 如果存在多个远程分支，优先选择提交时间范围内更活跃的分支。
- 如果仍无法区分，按分支名稳定排序后选择第一个，保证重复执行结果一致。

默认分支兜底规则：

- 优先使用仓库当前分支。
- 当前分支不可用时依次尝试 `origin/develop`、`origin/master`、`origin/main`、`develop`、`master`、`main`。
- 仍不可用时归类为 `未识别分支`。

## 提交去重规则

- 以完整 commit hash 作为唯一键。
- 同一项目内同一 commit 只统计一次。
- 不同项目即使 hash 相同，也按不同项目分别统计。

## 变更统计规则

- 每个 commit 统计变更文件数、新增行数、删除行数。
- 二进制文件不计入新增/删除行数，但计入变更文件数。
- rename 文件按 Git numstat 结果记录。
- 合并提交如无文件变更，允许显示 `未检测到文件变更`。

## 项目统计规则

项目维度至少统计：

- 提交次数。
- 新增行数。
- 删除行数。
- 文件改动次数。
- 唯一文件数。
- 活跃提交天数。

## 提交分类规则

提交分类优先根据 commit message，其次根据路径和文件类型辅助判断。

| 类型 | 识别关键词或特征 | 标准表述 |
| --- | --- | --- |
| 功能开发/优化 | `feat`、`优化`、`调整`、新增页面、接口对接 | 功能开发与细节优化 |
| 缺陷修复 | `fix`、`修复`、`bug`、`报错`、`问题处理` | 缺陷修复与稳定性处理 |
| 发布交付 | `打包`、`发版`、`发布`、`dist`、构建产物 | 版本打包与环境发布处理 |
| 国际化 | `国际化`、`i18n`、`文案`、语言包 | 国际化文案与界面展示优化 |
| 权限/字段隐藏 | `权限`、`隐藏`、`展示`、字段控制 | 字段权限与页面可见性收敛 |
| 合并同步 | `Merge branch`、`merge` | 分支合并与代码同步 |
| 样式/交互 | `样式`、`布局`、`弹窗`、`按钮`、`图标` | 页面布局与交互优化 |

## 主题归纳规则

- 优先按业务模块归纳，而不是按代码目录机械归纳。
- 同一项目内相近 commit 应合并为更高层主题。
- 主题名称应稳定，例如：`订单审核流程与审批交互完善`、`流程配置与流程看板能力增强`、`附件与下载链路优化`。
- 无法识别时使用：`功能开发与细节优化`。

## 文档结构规则

月报固定采用以下结构：

```markdown
# {YYYY} 年 {M} 月 Git 工作汇总

## 统计说明

## 第一部分：提交明细

### 项目：{project}

#### 分支：{branch}

## 第二部分：当月工作内容总结

### 事实版总结

### KPI 量化版总结

## 项目分布
```

## Commit Hash 展示规则

- 提交明细必须展示 commit hash。
- 默认展示短 hash，长度 7 到 8 位。
- Markdown 表格中使用反引号包裹，例如：`` `6513e1c` ``。

## 总结表达规则

- 事实版总结强调客观事实，不做绩效包装。
- KPI 量化版总结强调交付规模、覆盖范围、重点项目和核心业务线。
- 总结必须和项目分布、提交明细一致。
- 不使用无法由 Git 数据证明的表述，例如“上线成功”“用户体验显著提升”“业务收益明显增长”。

## 安全规则

- 不在配置、Prompt、日志或生成文档中写入 API Key、Token、密码等敏感信息。
- 如果提交信息中包含疑似敏感信息，生成文档时只保留必要上下文，并提示人工复核。
- 调用 AI 接口时只发送生成总结所需的最小数据集。
